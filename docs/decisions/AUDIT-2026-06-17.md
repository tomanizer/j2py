## Executive summary

j2py is a **well-governed beta with production-grade engineering hygiene and an honest,
well-understood correctness gap.** Engineering discipline is genuinely strong — strict gates pass
clean, the backlog is tightly curated, docs are in sync. The central risk is unchanged from prior
audits but now **quantified**: the translator achieves ~98–100% mechanical node coverage on dense
real-library samples while only **~29% of the public method surface is proven semantically
equivalent**. The project is honest about this. The newer concern is **corpus-overfitting creeping
into the rule layer** ([#356](https://github.com/tomanizer/j2py/issues/356)).

**Positioning:** A structurally faithful Java-to-Python translator with strong mechanical coverage
and explicit review checkpoints — not yet a correctness-verified porting tool at library scale.

## Findings (ranked)

### 1. Correctness proof remains narrow — the defining risk

Node coverage of ~98–100% measures **mechanical completion, not correctness**. A file at
`coverage == 1.0` can still emit silently-wrong semantics (operator precedence, bare static calls,
overload stubs). Only **28.9%** of the public surface is proven equivalent. CI now uploads
`equivalence-surface.json` and enforces the committed ratchet floor, so the remaining risk is the
size of the unverified surface rather than a passive metric. The gap between "looks complete" and
"proven equivalent" is the project's defining limitation.

### 2. Corpus-overfitting in the rule layer

`overloads.py` is now the largest module at **1,684 LOC** (up from 1,065 at the prior audit) and
contains class-specific dispatchers hardcoded to corpus fixtures:

- `_static_object_name_factory_dispatch_overload` — matches class `ObjectName` + method
  `get_instance` (JMX), backed by an `ObjectName` / `MalformedObjectNameException` stub in
  `j2py/translate/runtime/j2py_runtime.py`.
- `_char_string_append_dispatch_overload` — `StringBuilder`-specific.

These inflate dispatch/coverage metrics with logic that does not generalize. The maintainers track
this in [#356](https://github.com/tomanizer/j2py/issues/356), but it ships in the current beta.
This is "teaching to the test" that makes corpus scoreboards read better than the general capability
warrants — it should be flagged explicitly in any enterprise-fit assessment.

### 3. Coverage and equivalence floors now exist, but remain young gates

`pyproject.toml` sets `fail_under = 90`, the required `coverage` CI job runs `make test-cov`, and
the Python 3.11 test job enforces `scripts/equivalence/check_surface_floor.py` against the uploaded
`equivalence-surface.json`. These gates are meaningful, but branch coverage is still tracked only
as a reported metric (81.5%), not as a ratcheting floor. In a translator, branches are the construct
variants most worth covering.

### 4. CI marker parity needs to stay guarded

CI and `make check` now both exclude `behavior`, `live_llm`, and `target_translation` from the
normal test gate. Keep this parity explicit when adding future marker families so strict-xfail
roadmap targets do not accidentally enter one gate but not the other.

### 5. Dependency weight — resolved

At baseline `google-genai` was a hard install dependency despite being lazy-imported everywhere
(`j2py/llm/client.py:95`, `:261`), pulling a large transitive tree onto every Anthropic-only user.
Resolved shortly after this audit by #369 (#278): `google-genai` is now the optional `gemini`
extra (`pip install "j2py-converter[gemini]"`) and out of the default dependency set.

## Prioritized follow-up

### P0 — Strategic (correctness + gate integrity)

1. **Keep ratcheting the equivalence-verified surface.** The committed CI floor is now in place;
   raise it whenever new literal-oracle tests add passing public methods.
2. **Add a branch-coverage floor.** Line coverage is enforced, but branch coverage is still only
   reported despite being the better proxy for construct-variant coverage.

### P1 — Generality / governance

3. **Execute [#356](https://github.com/tomanizer/j2py/issues/356):** replace the `ObjectName` /
   char-append one-offs with a general erased-signature dispatcher. Until then, annotate them as
   known overfit in the corpus scoreboard so the numbers are not over-read.
4. **Keep CI marker filters aligned** with local `addopts` as new marker families are added.

### P2 — Packaging / maintainability

5. **Keep optional provider dependencies out of the default install.** `google-genai` is now in the
   `gemini` extra; preserve that split for future provider SDKs.
6. **Watch `overloads.py` (1.68k LOC) and `expr_ops.py` (1.0k LOC)** as the next refactor targets.
